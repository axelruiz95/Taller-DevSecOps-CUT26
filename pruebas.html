Autenticación y gestión de usuarios

ejercicio1/login.py Crítico

Vulnerabilidades encontradas

  • 🔴
    SQL Injection (x3) — Las funciones get_user, reset_password y create_user construyen consultas SQL con f-strings directamente desde input del usuario. Un atacante puede inyectar ' OR '1'='1 para eludir la autenticación o destruir la base de datos.
  • 🔴
    Credenciales hardcodeadas — DB_PASSWORD = "admin123" y SECRET_KEY = "mysecretkey" están en texto plano en el código fuente y aparecerán en el historial de Git para siempre.
  • 🔴
    Contraseña en texto plano — user[2] == password compara la contraseña recibida contra el valor almacenado en la base de datos sin hash, lo que significa que las contraseñas se guardan en claro.
  • 🟠
    Contraseña expuesta en logs — print(f"Password reset for user {user_id}: {new_pass}") registra la nueva contraseña en el log del sistema.
  • 🟡
    Resource leak — Varias conexiones sqlite3.connect() no se cierran explícitamente, lo que puede causar bloqueos de base de datos bajo carga.

Mejoras recomendadas

  • ✅
    Usar consultas parametrizadas: conn.execute("SELECT * FROM users WHERE username=?", (username,))
  • ✅
    Almacenar secretos en variables de entorno o un gestor de secretos (HashiCorp Vault, AWS Secrets Manager).
  • ✅
    Hashear contraseñas con bcrypt o argon2: bcrypt.checkpw(password.encode(), stored_hash)
  • ✅
    Nunca loguear datos sensibles. Registrar solo el user_id.
  • ✅
    Usar context managers (with sqlite3.connect(...) as conn:) para garantizar el cierre de conexiones.

Código corregido (fragmento)

import sqlite3, bcrypt, os

def get_user(username: str, password: str) -> dict:
    with sqlite3.connect(os.environ["DATABASE_URL"]) as conn:
        cursor = conn.execute(
            "SELECT id, username, password_hash FROM users WHERE username=?",
            (username,)
        )
        user = cursor.fetchone()

    if user and bcrypt.checkpw(password.encode(), user[2]):
        return {"status": "ok", "user": {"id": user[0], "username": user[1]}}
    return {"status": "error", "message": "Invalid credentials"}

Procesador de pagos

ejercicio2/payment.js Code Smell Bug

Problemas encontrados

  • 🔴
    NullPointerException — chargeCard pasa null como primer argumento a processPayment, que al no encontrar user != null, devuelve undefined. La línea siguiente result.total lanza un error de tipo en runtime.
  • 🟠
    Complejidad cognitiva excesiva — Cuatro niveles de if anidados en processPayment superan el umbral recomendado de Sonar (por defecto 15). El código es difícil de leer, testear y mantener.
  • 🟠
    Código duplicado — Los bloques de credit y debit son idénticos (misma fórmula de comisión * 0.03). Cualquier cambio en la comisión debe hacerse en dos lugares.
  • 🟡
    Igualdad laxa (== vs ===) — Comparaciones con == en JavaScript permiten coerciones de tipo inesperadas: "0" == false es true.
  • 🟡
    Sin return explícito en fallo — Si ninguna condición se cumple, la función devuelve undefined silenciosamente en lugar de un error descriptivo.

Mejoras recomendadas

  • ✅
    Extraer la lógica de comisión a una función applyFee(total) para eliminar la duplicación.
  • ✅
    Usar cláusulas de guarda (early return) para aplanar el anidamiento.
  • ✅
    Reemplazar todos los == por ===.

Código corregido (fragmento)

function applyFee(total) {
  return total + total * 0.03;
}

function processPayment(user, amount, type, currency, discount, tax) {
  if (!user || amount <= 0) return { status: "error", message: "Invalid input" };
  if (type !== "credit" && type !== "debit") return { status: "error", message: "Unknown type" };
  if (currency !== "MXN" && currency !== "USD") return { status: "error", message: "Unknown currency" };

  let total = amount - (discount ?? 0) + (tax ?? 0);
  total = applyFee(total);
  return { status: "ok", total, type, currency };
}

Procesador de archivos

ejercicio3/file_processor.py Crítico

Vulnerabilidades encontradas

  • 🔴
    Path Traversal — save_file(filename, content) acepta rutas como ../../etc/passwd. Aunque se use os.path.join, si filename empieza con / o contiene ../, el atacante puede escribir en cualquier directorio del servidor.
  • 🔴
    Command Injection — list_user_files inserta directamente el input del usuario en os.system(). Un valor como ; rm -rf / ejecutaría comandos destructivos en el servidor.
  • 🔴
    XXE (XML External Entity) — ET.fromstring(xml_string) procesa entidades XML externas por defecto en versiones antiguas de Python. Un atacante puede leer archivos del sistema o realizar SSRF mediante entidades como &xxe;.
  • 🟠
    Hash débil (MD5) — MD5 está considerado criptográficamente roto desde 2004. No debe usarse para verificar integridad de archivos críticos ni como identificador seguro.

Rules esperadas

Mejoras recomendadas

  • ✅
    Sanitizar el nombre de archivo: usar os.path.basename(filename) y verificar que la ruta resuelta empiece con UPLOAD_DIR.
  • ✅
    Reemplazar os.system() por subprocess.run(["ls", safe_path], capture_output=True) con lista de argumentos (sin shell).
  • ✅
    Para XML, usar defusedxml: import defusedxml.ElementTree as ET.
  • ✅
    Reemplazar MD5 por SHA-256: hashlib.sha256(content.encode()).hexdigest().

Código corregido (fragmento)

import hashlib, os, subprocess
import defusedxml.ElementTree as ET

def save_file(filename: str, content: str) -> str:
    safe_name = os.path.basename(filename)
    filepath  = os.path.join(UPLOAD_DIR, safe_name)
    if not filepath.startswith(UPLOAD_DIR):
        raise ValueError("Invalid filename")
    with open(filepath, "w") as f:
        f.write(content)
    return filepath

def get_file_hash(content: str) -> str:
    return hashlib.sha256(content.encode()).hexdigest()

def list_user_files(subdirectory: str):
    safe_sub = os.path.basename(subdirectory)
    subprocess.run(["ls", os.path.join(UPLOAD_DIR, safe_sub)], check=True)

Cliente de API interna

ejercicio4/api_client.py Crítico

Vulnerabilidades encontradas

  • 🔴
    Deserialización insegura (pickle) — pickle.loads(data) ejecuta código Python arbitrario si el payload está controlado por el atacante. Es una de las vulnerabilidades más graves en Python.
  • 🔴
    Credenciales hardcodeadas — API key, URL base y contraseña de admin están embebidas en el código fuente. Cualquier persona con acceso al repositorio tiene acceso al sistema.
  • 🔴
    Verificación SSL desactivada — verify=False en todas las llamadas HTTP desactiva la validación del certificado TLS, exponiendo la comunicación a ataques de tipo Man-in-the-Middle.
  • 🟠
    Excepción demasiado amplia — except Exception as e captura todos los errores posibles (incluyendo KeyboardInterrupt, errores de red, errores de parseo) y devuelve None, ocultando el problema real.
  • 🟠
    Escritura en /tmp sin validación — Los archivos .pkl se guardan en /tmp con nombres predecibles basados en la clave, lo que puede facilitar un ataque de symlink.

Rules esperadas

Mejoras recomendadas

  • ✅
    Reemplazar pickle por JSON o MessagePack para serialización de datos no confiables.
  • ✅
    Leer secretos desde variables de entorno: os.environ["API_KEY"]. Nunca commits de .env.

Perfil de usuario (frontend)

ejercicio5/user_profile.js Crítico Code Smell

Vulnerabilidades encontradas

  • 🔴
    XSS (Cross-Site Scripting) — innerHTML = userData.bio y innerHTML = "<h1>" + userData.name insertan HTML sin sanitizar. Un atacante puede inyectar <script>document.cookie</script> para robar sesiones.
  • 🔴
    Ejecución de código arbitrario (eval) — eval(code) ejecuta cualquier JavaScript, tanto del servidor como del usuario. Es una puerta de entrada directa para inyección de código.
  • 🔴
    Prototype Pollution — mergeOptions itera sobre todas las claves de source sin filtrar __proto__ o constructor. Un atacante puede contaminar el prototipo global de Object y afectar toda la aplicación.
  • 🟠
    Variable local no utilizada — let unused = score * 2 se declara pero nunca se usa, lo que indica código incompleto o un bug.
  • 🟡
    Función sin return implícito — getDiscount no tiene un return por defecto, devolviendo undefined si el usuario no es premium ni tiene créditos en cero.

Rules esperadas

Mejoras recomendadas

  • ✅

function renderProfile(userData) {
  document.getElementById("profile").textContent = userData.bio;
  const h1 = document.createElement("h1");
  h1.textContent = userData.name;
  document.getElementById("username").replaceChildren(h1);
}

function mergeOptions(target, source) {
  const FORBIDDEN = new Set(["__proto__", "constructor", "prototype"]);
  for (const key of Object.keys(source)) {
    if (!FORBIDDEN.has(key)) target[key] = source[key];
  }
  return target;
}

Generación de tokens de autenticación

ejercicio6/auth_token.py Crítico

Vulnerabilidades encontradas

  • 🔴
    PRNG inseguro — random.randint() no es criptográficamente seguro. Un atacante que conozca el estado del generador puede predecir todos los tokens futuros. Para seguridad usar secrets.
  • 🔴
    Seed predecible — random.seed(user_id) hace que el token de reset sea completamente determinista: conocer el user_id (dato público en muchos sistemas) es suficiente para calcular el token.
  • 🔴
    SSRF / Open Redirect — urllib.request.urlopen(url) sin validación permite que el atacante haga que el servidor realice peticiones a recursos internos (http://169.254.169.254/ en AWS) o a servicios externos.
  • 🟠
    TOCTOU (Race Condition) — os.path.exists(path) y open(path, "w") no son atómicos. Entre las dos llamadas, otro proceso puede crear o eliminar el archivo, causando comportamiento impredecible.
  • 🟠
    Timing Attack — token == sig usa comparación de strings en tiempo variable (Python termina al primer carácter diferente), lo que permite a un atacante medir tiempos de respuesta para adivinar el token carácter a carácter.

Reglas SonarQube que se activan

Rules esperadas

python:S2245 python:S5144 python:S4036 python:S2755

Mejoras recomendadas

  • ✅
    Usar secrets.token_hex(32) para todos los tokens de autenticación.
  • ✅
    Validar la URL contra una whitelist de dominios permitidos antes de realizar la petición.
  • ✅
    Usar open(path, "x") (modo exclusivo) en lugar del patrón exists+open para evitar TOCTOU.
  • ✅
    Usar hmac.compare_digest(token, sig) para comparaciones de tokens en tiempo constante.

Servicio JWT y cifrado

ejercicio7/jwt_service.js Crítico

Vulnerabilidades encontradas

  • 🔴
    JWT sin verificación de firma — verifyToken solo comprueba que el token tenga 3 partes y decodifica el payload directamente, sin verificar nunca la firma HMAC. Cualquier token manipulado sería aceptado como válido.
  • 🔴
    Secreto JWT hardcodeado — JWT_SECRET = "hardcoded_jwt_secret_123" está en el código fuente. Un atacante puede firmar sus propios tokens con rol admin.
  • 🔴
    Criptografía débil (DES-ECB) — DES tiene clave de 56 bits (rompible por fuerza bruta) y el modo ECB no produce cifrado semánticamente seguro (bloques idénticos producen cifrado idéntico).
  • 🟠
    HMAC con MD5 — MD5 está roto criptográficamente. Para HMAC de JWT usar SHA-256 como mínimo.
  • 🟠
    Token completo en logs — logAccess imprime el token JWT completo. Cualquiera con acceso a los logs puede usar ese token para autenticarse.

Reglas SonarQube que se activan

Rules esperadas

javascript:S2068 javascript:S4790 javascript:S5542 javascript:S4423 javascript:S5547

Mejoras recomendadas

  • ✅
    Usar una librería JWT establecida como jsonwebtoken que maneje la verificación de firma correctamente.
  • ✅
    Cargar el secreto desde variable de entorno: process.env.JWT_SECRET.
  • ✅
    Reemplazar DES-ECB por AES-256-GCM, que ofrece cifrado autenticado y es el estándar actual.
  • ✅
    En logs, registrar solo el userId y la fecha, nunca el token completo.

Código corregido (fragmento)

const jwt = require("jsonwebtoken");

function createToken(userId, role) {
  return jwt.sign(
    { userId, role },
    process.env.JWT_SECRET,
    { expiresIn: "24h", algorithm: "HS256" }
  );
}

function verifyToken(token) {
  return jwt.verify(token, process.env.JWT_SECRET);
}

function logAccess(userId) {
  console.log(`User ${userId} authenticated at ${new Date().toISOString()}`);
}

Generador de reportes

ejercicio8/report_generator.py Crítico

Vulnerabilidades encontradas

  • 🔴
    SSRF (Server-Side Request Forgery) — requests.get(url) sin validación permite que el servidor realice peticiones a URLs internas como http://169.254.169.254/latest/meta-data/ (metadata de AWS/GCP) o servicios en la red privada.
  • 🔴
    Template Injection — template.format(**user_data) con datos del usuario permite que un atacante use claves como {__class__.__init__.__globals__} para leer variables de entorno o ejecutar código.
  • 🔴
    Path Traversal en eliminación — "/reports/" + path sin normalizar permite eliminar archivos fuera del directorio de reportes.
  • 🟠
    TOCTOU en archivo temporal — tempfile.mktemp() solo genera un nombre, no crea el archivo. Entre la generación del nombre y la apertura con open(), otro proceso puede crear un symlink malicioso.
  • 🟠
    Permisos excesivos — os.chmod(tmp, 0o777) da acceso de lectura, escritura y ejecución a todos los usuarios del sistema sobre el reporte generado.
  • 🟡
    Log Injection — logging.info(f"User {user} requested: {url}") con input del usuario puede falsificar entradas de log si el usuario incluye saltos de línea.

Servicio de búsqueda

ejercicio9/search_service.js Crítico Code Smell

Vulnerabilidades encontradas

  • 🔴
    NoSQL Injection — buildMongoQuery inserta el input del usuario directamente en el objeto de query de MongoDB. Si el atacante envía { "$ne": null } en lugar de un string, la query se convierte en "todos los usuarios cuyo username no sea null", omitiendo cualquier filtro de autenticación.
  • 🔴
    ReDoS (Regular Expression Denial of Service) — El patrón /^([a-zA-Z0-9]+)*@/ tiene cuantificadores anidados. Una cadena como aaaaaaaaaaaaaaaa@ puede causar retroceso exponencial y bloquear el hilo de Node.js durante segundos o minutos.
  • 🟠
    Paginación sin límite — paginateResults ignora completamente size y MAX_RESULTS. Un atacante puede pedir la página 0 con size 0 y obtener todos los registros de la colección.
  • 🟡
    Condición muerta (dead code) — code === "SAVE10" && code === "SAVE20" es lógicamente imposible: una variable no puede ser dos valores distintos simultáneamente. Este bloque nunca se ejecuta.
  • 🟡
    Variable no utilizada — let unused = data.length se declara y nunca se referencia.

Cargador de configuración

ejercicio10/config_loader.py Crítico

Vulnerabilidades encontradas

  • 🔴
    Ejecución de código arbitrario (yaml.load) — yaml.load(yaml_string) sin un Loader seguro puede ejecutar código Python arbitrario mediante objetos YAML especiales como !!python/object/apply:os.system ["rm -rf /"].
  • 🔴
    LDAP Injection — f"(uid={username})" sin escape permite a un atacante inyectar filtros LDAP arbitrarios. Por ejemplo, *)(uid=*))(|(uid=* puede autenticar como cualquier usuario.
  • 🔴
    Command Injection — "tar -czf backup.tar.gz " + directory con shell=True permite ejecutar comandos adicionales si directory contiene ; rm -rf /.
  • 🔴
    Credencial hardcodeada — SECRET_TOKEN con un valor JWT en el código fuente es accesible a cualquiera con acceso al repositorio.
  • 🟠
    Hash débil para contraseñas (SHA-1 sin sal) — SHA-1 es vulnerable a colisiones y, sin sal, susceptible a ataques de diccionario y rainbow tables. Las contraseñas hasheadas con SHA-1 pueden romperse rápidamente con herramientas como Hashcat.
  • 🟡
    Función muerta — dead_function() nunca es invocada en el módulo. Código no alcanzable que aumenta la deuda técnica.

Mejoras recomendadas

  • ✅
    Usar yaml.safe_load(yaml_string) que solo permite tipos YAML básicos y nunca ejecuta código.

import yaml, subprocess, bcrypt, os
from ldap3.utils.conv import escape_filter_chars

def load_config(yaml_string: str) -> dict:
    return yaml.safe_load(yaml_string)

def search_user(ldap_conn, username: str):
    safe_username = escape_filter_chars(username)
    search_filter = f"(uid={safe_username})"
    return ldap_conn.search_s("dc=corp,dc=com", 2, search_filter)

def run_backup(directory: str):
    subprocess.run(
        ["tar", "-czf", "backup.tar.gz", directory],
        check=True
    )

def hash_password(password: str) -> bytes:
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt())
